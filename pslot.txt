# Parking Slot Management System – Owner-Based Approach

## 1. Project Idea

Build a **Parking Slot Management System** where not only admins but also **parking owners (shop owners, mall owners, building owners, etc.)** can register their parking spaces.
Users can then **view available parking locations and book slots**.

This approach makes the system similar to a **parking marketplace platform**.

---

# 2. System Roles

### 1. User

Regular users who want to park their vehicles.

Capabilities:

* Register / Login
* View available parking locations
* View available parking slots
* Book a parking slot
* Cancel booking

---

### 2. Parking Owner

Owners who want to list their parking spaces in the system.

Examples:

* Mall owners
* Shop owners
* Apartment owners
* Hotel owners

Capabilities:

* Register as owner
* Add parking location
* Add parking slots
* View bookings for their parking
* Manage parking availability

---

### 3. Admin

Admin controls the system.

Capabilities:

* Approve parking locations added by owners
* Manage users
* View all bookings
* View system statistics
* Disable or remove parking locations

---

# 3. System Flow

Owner Registration Flow:

Owner registers
→ Adds parking location
→ Adds parking slots
→ Admin reviews parking
→ Admin approves parking
→ Parking becomes visible to users

User Booking Flow:

User registers / logs in
→ Views parking locations
→ Selects a location
→ Views available slots
→ Books a slot
→ Slot status becomes **booked**

---

# 4. Database Design

### User Model

Stores all users, owners, and admins.

Fields:

* name
* email
* password
* role (user | owner | admin)

---

### ParkingLocation Model

Represents a parking area.

Fields:

* name
* address
* totalSlots
* ownerId
* status (pending | approved)

---

### ParkingSlot Model

Each parking location contains multiple slots.

Fields:

* slotNumber
* locationId
* status (available | booked | occupied)

---

### Booking Model

Stores user parking bookings.

Fields:

* userId
* slotId
* startTime
* endTime
* status (booked | completed | cancelled)

# 6. Example API Structure

Owner APIs:

* POST /api/parking/add
* GET /api/parking/my-parking
* POST /api/parking/add-slot

User APIs:

* GET /api/parking
* GET /api/slots/:locationId
* POST /api/booking

Admin APIs:

* GET /api/admin/pending-parkings
* PUT /api/admin/approve-parking
